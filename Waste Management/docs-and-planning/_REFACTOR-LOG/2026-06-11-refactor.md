## Architecture

```
Client request
     ↓
legacyTemplateAiRoutes.adapter.js  (mounts all 7 step routes)
     ↓
steps/0N-STEP-NAME/route.js        (express router, applies feature flags + auth)
     ↓
steps/0N-STEP-NAME/controller.js   (thin HTTP wrapper)
     ↓
server/src/services/               (all business logic)
     ↓
steps/07-HANDOFF-DEPLOY/deploy.stage.js   (pre-flight + GitHub push)
     ↓
01-HOSTING-DEPLOY-ENGINE           (existing GitHub → Render pipeline — unchanged)
     ↓
Deployment Settings UI             (repo URL pre-filled + read-only, client deploys themselves)
```

## OpenAI Flow

```
Client JSON (brief answers)
     ↓
openaiTemplateAssistant.service.js:buildPayloadZip()
  → ZIP: all master template files + client.json injected
     ↓
sendToOpenAI()
  → Upload ZIP to OpenAI Files API
  → Create Assistant with Code Interpreter
  → Create Thread + Run
     ↓
pollForCompletion()
  → Poll run.status === 'completed'
     ↓
downloadResultZip()
  → Download tailored-site.zip from OpenAI
     ↓
deploy.stage.js
  → Unzip scan → sensitive file check → push to GitHub
     ↓
01-HOSTING-DEPLOY-ENGINE → Render
```

## Security Notes
- No .env files, API keys, GitHub tokens, Render tokens, or private keys are ever committed to GitHub
- deploy.stage.js blocks deployment if sensitive file patterns are detected
- OPENAI_API_KEY read from process.env only — never logged or exposed
- Secrets passed to Render as environment variables via the existing hosting engine
